## Intent

Add the repository-side automation needed for the release-governance model:

```text
feature/* -> stg -> main
```

The repository currently uses `main` as its production branch
(`origin/HEAD -> origin/main`), so the production workflow targets `main`, not
`master`.

## Design

- CI uses GitHub-hosted `ubuntu-latest` because it only needs source validation.
- Deploy jobs use a self-hosted runner on the deployment host, so deployment can
  access the local Docker API without exposing Docker or SSH credentials to
  GitHub-hosted workers.
- Staging and production workflows generate gitignored env files from GitHub
  Environment variables, then call the same shared deployment script
  introduced in the staging Phase 1 work.
- Staging uses the host-managed `cloudflared.service` route
  `stg.t3s7.com -> http://127.0.0.1:3001`, so the default GitHub workflow does
  not require `STAGING_TUNNEL_TOKEN`.
- The production tunnel URL and staging Cloudflare dashboard route are treated
  as external platform configuration; repository workflows deploy app
  containers and do not own either tunnel token.
- Production workflow requires `PRODUCTION_SITE_URL` instead of silently using a
  localhost metadata base in automation.
- Production workflow has a `require-staging` job that runs before the
  production environment deployment. It requires the exact production
  `GITHUB_SHA` to have a successful deployment status in the `staging`
  environment.

## Required External Setup

Before branch automation can actually deploy:

- Register a GitHub self-hosted runner on the target host.
- Add runner label `fuma-lab`.
- Ensure the runner user can access Docker and Compose.
- Create GitHub Environments `staging` and `production`.
- Confirm the host Cloudflare tunnel routes `stg.t3s7.com` to
  `http://127.0.0.1:3001`.
- Set `PRODUCTION_SITE_URL` as a production variable.
- Configure required reviewers on the `production` Environment if product
  approval should gate production deployment.
- Promote the exact staged SHA to `main`. A merge commit with a new SHA will not
  satisfy the staging deployment gate unless that new SHA is deployed to staging
  first.
- Use the checklist in `docs/features/fumadocs/release-automation.md` for
  staging product acceptance before approving production.
