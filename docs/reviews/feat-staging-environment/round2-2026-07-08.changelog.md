## Intent

Refactor the staging work from a separate manual lifecycle into the first
implementation slice of the release-governance model:

```text
local/dev -> staging -> production
```

Local remains fast and disposable. Staging and production now share one
deployment implementation with environment-specific configuration, image tags,
health checks, smoke tests, and rollback behavior.

## Behavioral Model

- Local/dev:
  - Bare `docker compose up -d --build`.
  - Project `fuma-lab-local`.
  - Image `fuma-lab:local`.
  - Port `127.0.0.1:3009`.
- Production:
  - `npm run deploy` or `npm run deploy:prod`.
  - Project `fuma-lab`.
  - Image `fuma-lab:prod`.
  - Release tag `fuma-lab:production-<git-sha>-<timestamp>`.
  - Rollback tag `fuma-lab:production-rollback-<timestamp>`.
- Staging:
  - `npm run deploy:stg` or `scripts/deploy.sh --env staging`.
  - Project `fuma-lab-stg`.
  - Image `fuma-lab:stg`.
  - Release tag `fuma-lab:staging-<git-sha>-<timestamp>`.
  - Rollback tag `fuma-lab:staging-rollback-<timestamp>`.
  - Requires `ROBOTS_NOINDEX=1` and non-localhost `SITE_URL`.

## Safety Checks Added

- Production refuses `ROBOTS_NOINDEX=1`.
- Production refuses the placeholder production domain.
- Production project must be `fuma-lab`.
- Staging project must be `fuma-lab-stg`.
- Staging must build with `ROBOTS_NOINDEX=1`.
- Staging deploy refuses placeholder or missing tunnel token when the tunnel
  profile is enabled.
- Staging smoke checks require `X-Robots-Tag: noindex`.
- Production smoke checks reject accidental `noindex`.
